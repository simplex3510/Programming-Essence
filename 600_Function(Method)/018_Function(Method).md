# Funtion (함수)

프로그래밍에서 함수는 단순히 코드를 나누는 단위를 넘어, 효율적인 개발을 위한 핵심 도구이다.

함수를 사용함에 있어, 그 필요성을 크게 4가지로 나눌 수 있다.

1. **코드의 재사용성 (Reuse)**  
    똑같은 로직을 여러 번 쓸 때, 매번 복사해서 붙여넣는 대신 함수로 만들어 호출만 하면 된다.

    * C/C++: 표준 입출력이나 수학 계산을 `printf`, `scanf`, `sqrt` 같은 표준 함수로 호출해 사용한다.
    * C#: 메서드(Method) 형태로 정의하여 프로그램의 여러 곳에서 반복 호출한다.

2. **유지보수의 용이성 (Maintainability)**  
    코드에 수정 사항이 생겼을 때, 함수가 없다면 복사된 모든 곳을 찾아 고쳐야 한다.  
    하지만 함수를 사용하면 해당 함수 본체만 수정하면 된다.  

    * 예를 들어, 세금 계산 공식이 바뀌었을 때 `CalculateTax()` 함수 한 곳만 수정하면 프로그램 전체에 반영된다.

3. **구조화와 가독성 (Readability)**  
    수천 줄의 코드가 `main` 함수 하나에 다 들어있으면 해석하기가 매우 어렵다.  
    기능을 작은 단위의 함수로 쪼개면 전체적인 흐름을 한눈에 파악할 수 있다.

    * C/C++: 절차지향적 성격이 강해 기능을 단계별 함수로 나누어 관리한다.
    * C#: 객체의 행위를 메서드로 정의하여 논리적인 설계(OOP)를 가능하게 한다.

4. **메모리 효율과 모듈화 (Modularity)**  
    함수는 호출될 때만 메모리(스택)를 점유하고 처리가 끝나면 비워진다.  
    또한, 자주 쓰는 기능을 별도의 파일(.cpp, .cs)로 분리해 두면 다른 프로젝트에서도 쉽게 가져다 쓸 수 있는 '모듈'이 된다.

## Function Definition (함수 정의)

C#에서의 함수를 정의하는 문법은 다음과 같다.

```cs
[ReturnType] [FuncName]([Parameters])
{
    // Body of Function
}
```

이때 함수 정의에서는 다음의 것들을 반드시 포함해야 한다.

* Return Type
* *Function Signature
  * Function Name (필수)
  * Parameters (선택)
* Body of Function

> **Function Signature (함수 시그니처)**  
    함수 시그니처는 컴파일러가 함수를 식별(구분)하기 위해 사용하는 고유한 요소들이다.  
    이때, 포함되어야 하는 요소는 함수의 이름, 매개변수의 개수, 매개변수의 타입, 매개변수의 순서가 있다.  
    다만, 반환 타입은 시그니처에 포함되지 않음에 유의해야 한다.

가령, 두 정수의 합을 구하는 `Add` 함수를 다음과 같이 정의할 수 있다.

```cs
int Add(int num1, int num2)
{
    int result = num1 + num2;
    return result;
}
```

## Return Type (반환형)

반환형(Return Type)은 함수가 모든 작업을 마친 후, 자신을 호출한 곳으로 돌려주는 데이터의 종류(Data Type)를 의미한다.

이는 곧 함수의 출력(output)을 의미한다.

주요 반환형의 종류는 다음과 같다.

* Primitive Types (int, double, char, etc...)  
    계산 결과나 상태 값을 숫자나 문자로 반환한다.

* `void` Type  
    반환형이 `void`로 표현된 것은, "반환할 값이 없음"을 의미한다.

* Pointer/Reference Type
    메모리의 주소나 객체의 참조를 반환한다.

* Custom Type (struct, class)  
    직접 만든 복잡한 객체를 통째로 반환할 수도 있다.

C/C++에서 반환형이 지정된 함수는 반드시 `return`문으로 값을 반환해야 한다.  
그렇지 않으면 컴파일 에러가 발생한다.

C#도 마찬가지로 반환형이 지정된 함수는 반드시 값을 반환해야 한다.
또한, 컴파일러 측에서 모든 코드 경로에서 반드시 값이 반환되도록 더 엄격하게 검사한다.

## Parameters (매개변수)

매개변수(parameter)는 함수가 실행될 때 외부로부터 값을 전달받기 위해, 즉 입력으로써 사용된다.

매개변수는 함수에서 필수가 아니며, 따라서 함수는 0개 이상의 매개변수를 가질 수 있다.

`int`나 `char`와 같은 기본 자료형뿐만 아니라, `int[]`, `string[]`과 같은 배열도 매개변수로 사용이 가능하다.

### Arguments (인자)

인자는 함수를 호출할 때 함수로 전달하는 실제 값을 의미한다.

엄격하게 구분하자면 인자는 매개변수와 다르다.  
매개변수는 함수가 정의될 때 함수의 입장에서 전달되는 입력값을 의미한다.  
인자는 호출자의 입장에서 전달하는 입력값을 의미한다.

그러나 실질적으로 그 용도가 유사하기에 반드시 구분지어서 언급하지는 않으며, 둘을 혼용하여 사용하는 경우가 잦다.

## Body of Function (함수의 몸체)

함수 기능을 구현한 것이며, 중괄호(`{}`)로 영역을 구분짓고 내부에 코드가 존재한다.

만약 함수가 `void`가 아닌 반환형을 가지고 있다면, 반드시 `return` 키워드를 이용하여 적절한 타입의 데이터를 반환해야 한다.

함수의 반환형이 `void`이더라도 `return` 키워드만을 사용하여 명시적으로 함수의 종료를 나타낼 수도 있다.

## Function Name (함수 이름)

함수 이름의 명명법에 강제성이나 규칙은 없으나 기능과 용도에 적절한 이름으로 명명해야 한다.

예를 들어 `Add`라는 이름의 함수는 `A`라는 이름으로 정의한 것보다 더 명확하고, 무엇을 반환할지가 어느정도 예측이 가능하다.

## Function Call (함수 호출)

함수 호출은 어떤 함수를 사용하는 것을 의미한다.

예를 들어, 다음과 같은 코드가 있다고 가정한다.

```cs
static int Add(int num1, int num2)
{
    return num1 + num2;
}

static int Main(string[] args)
{
    int result = Add(1, 2);
}
```

함수 이름을 명시하여 `Add`라는 이름의 함수를 사용함을 알린다.  
해당 함수는 2개의 매개변수를 받으므로, `1`과 `2`라는 인자를 콤마(`,`)로 구분지어 전달한다.

이때, 매개변수의 개수를 다르게 전달하면 컴파일러는 다른 함수를 호출하는 것으로 인식하며, 그러한 함수가 없을 경우 컴파일 에러가 발생한다.  
왜냐하면, 함수의 시그니처가 달라서 고유한 다른 함수로 판단하기 때문이다.  

그러한 `Add` 함수는 평가되면 반환값이 `int`값인 `3`이 함수가 호출된 자리에 남는다.  
이때, `int` 타입인 `result` 변수가 이 `3`이라는 값을 저장하는 흐름으로 코드는 구성되어 있다.

### `void` 반환

반환형이 `void`인 함수는 별 다른 출력값이 없다.

따라서 다음과 같이 예를 들 수 있다.

```cs
static void PrintHello()
{
    Debug.Log("Hello!");
}

static void Main(string[] args)
{
    PrintHello();
}
```

`PrintHello` 함수는 매개변수를 받지도 않고, 값을 반환하지도 않는다.
단지 호출되었을 때, 프로그램의 실행 흐름이 `Main`에서 `PrintHello`로 이동, 다시 `Main`으로 복귀할 뿐이다.

### 인자 전달

함수를 호출할 때, 인자를 전달함에 있어서 변수와 상수(리터럴) 모두 가능하다.

즉, 다음과 같이 사용이 가능하다.

```cs
static int Add(int num1, int num2)
{
    return num1 + num2;
}

static int Main(string[] args)
{
    int num = 10;
    int result = Add(num, 100);
}
```

## 함수 정의를 유추하는 방법

가령 다음과 같은 함수 호출문이 있다고 가정한다.

```cs
double num = 10.0;
double result = Square(num);
```

위 코드를 보고 다음과 같은 요소들을 유추할 수 있다.

* 반환된 값이 `double`형 변수 `result`에 저장되므로, 반환형은 `double`임.
* 인자로 `double`형 변수 `num`에 저장되므로, 매개변수는 하나며 매개변수의 타입은 `double`임.
* 함수를 호출할 때 `Square`라는 이름을 사용하므로, 함수의 이름은 `Square`임.

이를 통해 다음과 같이 함수의 정의를 유추할 수 있다.

```cs
// 단, 매개변수의 이름까지는 유추할 수 없다.
double Square(double paramName);
```

함수의 이름을 통해서, 해당 함수의 기능까지도 추측이 가능하다.  
함수명이 `Square`이므로, 전달한 인자의 제곱을 구하는 것으로 추측할 수 있다.

하지만, 내부적인 구현은 무수히 많은 방식이 있을 수 있으므로, 구체적인 코드까지는 위 내용 만으로 알 수 없다.

## Black Box

함수는 반드시 블랙박스와 같아야 한다.
즉, 블랙박스와 같이 내부 구조를 알 수 없어야 한다.

호출자는 피호출자인 함수의 내부에 대해 알 수 없도록 하는 것이 좋다.

이를 위해서는 반환형, 함수명, 매개변수의 이름과 개수로 함수의 기능을 유추할 수 있도록 해야 한다.

```cs
// 첫 번째 예 : 좋은 예시
int Add(int op1, int op2)
{
    return op1 + op2;
}

// 두 번째 예 :나쁜 예시
int A(int a, int b)
{
    return op1 + op2;
}
```

어떤 프로그래머가 다른 프로그래머가 만든 함수를 사용하고자 할 때를 가정한다.

이때, 첫 번째 예시의 코드의 시그니처만 보아도 해당 함수가 어떤 기능을 하는지, 인자는 어떤 것을 받는지 등을 통해서 다음을 유추할 수 있다.

> 해당 함수는 이름이 `Add`이므로 `더하기` 기능을 가진 함수이다.  
    또한 2개의 `int`형 인자를 받으므로 매개변수로 전달받은 두 값을 더한다.  
    최종적으로 더한 값은 `int`형이고, 이 값을 반환한다.

이를 통해 프로그래머가 어떤 값을 전달하고(입력), 다시 어떤 값을 받을지(출력)에 대해서 적절히 활용할 수 있다.

반면, 두 번째 예시의 코드에서 시그니처는 그러한 단서를 제공하지 않으므로 사용자 프로그래머가 번거롭게 코드의 구현을 확인해야 하는 등의 불필요한 시간을 소모해야 한다.

즉, 가독성과 효율성을 위해서라도 함수 시그니처만으로도 해당 기능(구현)에 대해서 유추할 수 있어야 한다.

### Coding Standard : Function Naming Rule

함수의 이름은 다음과 같은 규칙을 지키는 것이 좋다.

* 해당 함수가 정확하게 어떤 기능을 하는지 알려주는 단어를 사용하여 표현
  * 함수는 행동 또는 행위를 의미하므로, 동사 단어로 시작
  * 첫 글자는 대문자로 시작
* 여러 단어를 연결하여 표현해야 한다면, 두 번째 단어부터는 첫 글자만 대문자로 표기

### Coding Standarad : Parameter and Local Variable Naming Rule

매개변수와 지역변수도 결국에는 변수다.
따라서 다음과 같은 규칙을 지키는 것이 좋다.

* 어떤 정보를 담는 것인지 명확하게 표현되는 단어 사용
  * 명사 사용
  * 첫 글자는 소문자로 시작
* 여러 단어를 연결하여 표현해야 한다면, 두 번째 단어부터는 첫 글자만 대문자로 표기

## Precondition(선조건) / Postcondition(후조건)

선조건과 후조건은, 함수가 어떤 동작을 하는지에 대한 약속을 의미한다.

선조건은, 함수 실행 시작 전에 참으로 가정한 조건을 의미한다.  
선조건에 대한 단적인 예시로, `Divide` 함수는 나누는 수(분모)가 `0`이 아니어야 한다.

선조건은 함수 이름이나 매개변수로 유추가 가능하나, 부족한 경우에는 주석으로 추가 설명을 하는 것이 좋다.

예를 들어, 다음과 같다.

```cs
/// <summary></summary>
/// <param name = "numerator"></param>
/// <param name = "denominator">no zero</param>
/// <returns></returns>
float Divide(float numerator, float denominator)
{
    return numerator / denominator;
}
```

이러한 주석을 문서 주석(Documentation Comment)이라고 한다.

---

반면 후조건은, 함수 실행 후에 보장되는 조건을 의미한다.  
후조건에 대한 단적인 예시로, 두 정수를 더하는 함수의 결과로 정수를 반환한다.

일반적으로는, 함수 이름과 반환형으로 유추가 가능하다.

---

선조건을 만족하지 못하면, 후조건을 보장할 수 없다.

선조건을 만족하지 못한 상태에서 진행된 함수는 정상적으로 동작하리라는 보장이 없다.  
따라서 정상적으로 동작하지 못한 함수가 후조건을 보장할 수 없는 것은 당연하다.

## 함수 시그니처의 약속

함수 시그니처에는 포함되지 않으나, 반환형을 포함하여 설명한다.

가령, 다음처럼 함수 시그니처가 표기되어 있을 때, 각 항목은 다음을 약속한다.

```cs
float Divide(float numerator, float denominator)
{
    return numerator / denominator;
} 
```

* 함수명: `Divide`
  * 어떤 수에서 다른 수를 나눈 결과를 얻을 수 있는 함수임

* 매개변수: `float numerator`, `float denominator`
  * 첫 번째 매개변수가 '분자'에 해당하고, 두 번째 매개변수가 '분모'에 해당함

* 반환형: `float`
  * 부동소수점 실수형이 반환됨

이렇듯 내부 구현이 몇 줄이 되고 얼마나 길든, 함수 내부를 모르더라도 함수의 시그니처와 반환형만을 보고 그 기능을 유추할 수 있도록 하는 것이 좋다.

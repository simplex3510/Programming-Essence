# 🎮 C# OOP 심화: 중첩·추상 클래스 및 인터페이스 문제 8선

### 1. 몬스터 드롭 아이템 구조 설계 (중첩 클래스)

특정 몬스터가 사망했을 때 드롭하는 아이템 정보는 해당 몬스터 클래스 내부에서만 긴밀하게 관리되는 것이 좋습니다. 이를 중첩 클래스로 구현하세요.

* **조건:**
* 외곽 클래스로 `Monster` 클래스를 정의하고, 내부에 `private` 접근 제한자를 가진 중첩 클래스 `DropItem`을 만드세요.
* 중첩 클래스 `DropItem`은 `string itemName`과 `int dropChance`(0~100) 필드를 가집니다.
* 외곽 클래스 `Monster` 내부에서 `DropItem` 객체를 생성하여 정보를 설정하는 메서드를 구현하세요.
* **생각해볼 점:** `DropItem` 클래스가 `private`으로 선언되었을 때, `Main` 메서드에서 `Monster.DropItem` 형태로 객체를 직접 생성할 수 없는 이유와 이를 통한 데이터 은닉의 이점을 서술하세요.

### 2. 기본 캐릭터의 뼈대 형성 (추상 클래스 기초)

모든 플레이어 직업군의 공통 분모를 모아둔 추상 클래스를 설계하고, 자식 클래스에서 이를 강제로 구현하도록 만드세요.

* **조건:**
* 추상 클래스 `Character`를 선언하세요.
* 모든 캐릭터가 공통으로 가질 일반 프로퍼티 `public string Name { get; set; }`을 선언하세요.
* 직업마다 공격 방식이 완전히 다르므로, 바디(`{ }`)가 없는 추상 메서드 `public abstract void Attack();`을 선언하세요.
* `Character`를 상속받는 자식 클래스 `Warrior`(전사)를 만들고, 부모의 추상 메서드를 오버라이딩(`override`)하여 `"대검을 휘두릅니다!"`가 출력되도록 구현하세요.

### 3. 아이템 사용 규칙 정의 (인터페이스 기초)

인터페이스는 클래스 계층 구조와 상관없이 "특정 행동을 할 수 있다"는 기능적 계약을 맺을 때 사용합니다.

* **조건:**
* 인터페이스 `IUsable`을 선언하세요. 내부에 반환 형이 없고 매개변수가 없는 `void Use();` 메서드를 정의하세요. (인터페이스 내부에는 접근 제한자를 붙이지 않습니다.)
* `Potion` 클래스와 `Scroll` 클래스를 만들고, 두 클래스 모두 `IUsable` 인터페이스를 구현(Implementation)하도록 하세요.
* `Potion`의 `Use()`는 `"포션을 마셔 HP를 회복합니다."`, `Scroll`의 `Use()`는 `"주문서를 사용해 장비를 강화합니다."`를 출력하게 만드세요.
* `Main`에서 두 객체를 생성하고 `Use()` 메서드를 각각 호출하세요.

### 4. 추상 클래스 vs 인터페이스 (종합 설계)

추상 클래스(개념적 상속, 단일 상속)와 인터페이스(기능적 포함, 다중 구현)의 결정적인 차이를 코드로 체감하는 문제입니다.

* **조건:**
* 부모 추상 클래스로 `Vehicle`(탈것)을 만들고, 추상 메서드로 `public abstract void Move();`를 두세요.
* 날 수 있는 기능을 뜻하는 인터페이스 `IFlyable`을 만들고, `void Fly();` 메서드를 두세요.
* `Vehicle`을 상속받으면서 동시에 `IFlyable`을 구현하는 **`Airship`(비행선) 클래스**를 완성하세요.
* **핵심 질문:** C#에서 `class Airship : Vehicle, ParentClass2` 처럼 두 개 이상의 클래스를 동시에 상속받는 것은 불가능하지만, 인터페이스는 여러 개를 붙일 수 있는 이유(다중 상속 제한)를 서술하세요.

### 5. 다형성 배열 인터페이스 확장 (아이템 일괄 사용)

서로 다른 가문(부모 클래스)에서 태어난 객체들이라도 같은 인터페이스를 입고 있다면 하나의 배열로 묶을 수 있습니다. (상속 캐스팅 응용)

* **조건:**
* 3번 문제에서 만든 `IUsable`, `Potion`, `Scroll` 클래스를 활용합니다.
* 여기에 전혀 다른 부모를 가질 수 있는 `Key` 클래스를 추가하고 이 클래스 역시 `IUsable`을 구현하게 하세요. (`Use()` 호출 시 "잠긴 문을 엽니다." 출력)
* `Main` 메서드에서 **`IUsable[] inventory = new IUsable[3];` 인터페이스 배열**을 선언하세요.
* 배열에 `Potion`, `Scroll`, `Key` 객체를 각각 담은 뒤, `for`문을 순회하며 모든 요소의 `Use()` 메서드를 일괄 호출하세요.

### 6. 인터페이스 안전한 형변환과 검사 (`is`, `as` 결합)

인터페이스 타입 배열에 담긴 객체 중, 특정 고유 기능을 가진 클래스만 골라내어 다운캐스팅하는 실무형 문제입니다.

* **조건:**
* 인터페이스 `IInteractable`(`void Interact();`)을 만드세요.
* 이를 구현하는 `Chest`(상자) 클래스와 `Npc` 클래스를 만드세요.
* `Chest` 클래스에는 인터페이스 메서드 외에 **`public void OpenWithKey()`라는 독자적인 메서드**가 추가로 존재합니다.
* `Main`에서 `IInteractable[] objects = new IInteractable[2];` 배열에 `Npc`와 `Chest`를 담으세요.
* `for`문으로 배열을 순회하며 `Interact()`를 호출하되, **만약 대상이 `Chest` 타입이면 `is` 또는 `as` 연산자로 다운캐스팅하여 `OpenWithKey()`까지 연쇄 호출**되도록 하세요.

### 7. 추상 클래스의 공통 로직과 추상 메서드의 조화

추상 클래스는 인터페이스와 달리 완전히 구현된 일반 메서드와 상태(필드)를 가질 수 있어, 중복 코드를 제거하는 데 탁월합니다.

* **조건:**
* 추상 클래스 `Spell`(마법)을 만드세요.
* 내부 필드로 `protected int manaCost;`를 두고, 일반 메서드로 `public void ConsumeMana()` (출력: "{manaCost}의 마나가 소목되었습니다.")를 구현하세요.
* 하위 클래스에서 마법 효과를 다르게 낼 수 있도록 추상 메서드 `public abstract void CastEffect();`를 선언하세요.
* `Spell`을 상속받는 `FireBall` 클래스를 만들고, 생성자에서 `manaCost`를 30으로 설정한 뒤 `CastEffect()`를 오버라이딩하여 화염구가 날아가는 연출을 출력하세요.
* `Main`에서 `FireBall`을 실행할 때, 부모의 `ConsumeMana()`와 자신의 `CastEffect()`가 차례로 실행되는 통합 메서드를 설계하세요.

### 8. 추상 클래스 내부의 중첩 클래스 (UI 시스템 구조)

게임 UI 창(Window) 시스템을 설계할 때 자주 사용되는 구조로, 추상 클래스와 중첩 클래스를 복합적으로 융합한 난이도 높은 문제입니다.

* **조건:**
* 추상 클래스 `GameWindow`를 만드세요. 내부에는 추상 메서드 `public abstract void Draw();`가 있습니다.
* `GameWindow` 클래스 내부에 `public` 접근 제한자를 가진 중첩 클래스 `WindowRect`(창의 좌표 및 크기 변수 그룹)를 정의하세요. `WindowRect`는 `int x, y, width, height;` 필드를 가집니다.
* `GameWindow`를 상속받는 자식 클래스 `InventoryWindow`를 만드세요.
* `Main` 메서드에서 `InventoryWindow` 인스턴스를 생성하고, 외곽 클래스를 통해 내부 중첩 클래스 타입의 변수를 선언 및 활용하는 코드를 작성하세요. (예: `GameWindow.WindowRect rect = ...`)



---

* **1번, 8번 (중첩 클래스):** 외부 클래스와 내부 클래스의 접근 제한자 관계(`private`과 `public`에 따른 범위 차이)와 타 클래스에서 호출 시 `외곽클래스.중첩클래스` 문법을 정확히 인지했는지 평가합니다.
* **2번, 7번 (추상 클래스):** 추상 클래스는 `new` 키워드로 **스스로 인스턴스를 생성할 수 없다**는 점과, 추상 메서드는 자식에서 무조건 `override`해야 컴파일 에러가 나지 않는다는 강제성을 이해시킵니다.
* **3번, 5번, 6번 (인터페이스):** 클래스 상속 구조에 얽매이지 않고 여러 클래스에 공통 규격(`IUsable`, `IInteractable`)을 입혀 다형성을 다채롭게 사용하는 능력을 극대화합니다.
